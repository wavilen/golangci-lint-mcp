# staticcheck: SA2003

<instructions>
Detects a mutex `Lock` call whose `Unlock` is deferred, but `Lock` itself is not deferred. This means if the code between `Lock` and `defer Unlock` panics, `Unlock` still runs correctly — but the real issue is when the `defer Unlock` appears before the `Lock`, causing `Unlock` to be called on an unlocked mutex (undefined behavior).
</instructions>

<examples>
## Bad
```go
defer mu.Unlock() // scheduled first
mu.Lock()         // lock acquired after Unlock is already deferred
doWork()
```

## Good
```go
mu.Lock()
defer mu.Unlock() // deferred after Lock
doWork()
```
</examples>

<patterns>
- Move `defer mu.Unlock()` below `mu.Lock()` — never place the deferred unlock before the lock acquisition
- Fix copy-paste errors where `defer` and `Lock` order is swapped
- Place `defer mu.Unlock()` immediately after `mu.Lock()`, not at the function top
</patterns>
