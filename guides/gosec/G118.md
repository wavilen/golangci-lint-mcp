# G118

<instructions>
Detects context propagation failures that lead to goroutine or resource leaks. Covers three patterns: (1) `context.WithCancel`/`WithTimeout`/`WithDeadline` where the cancel function is never called, (2) goroutines using `context.Background()` when a request-scoped context is available, and (3) infinite loops performing blocking I/O without checking `ctx.Done()`.

Always `defer cancel()` when creating derived contexts. Propagate request contexts into goroutines. Check `ctx.Done()` in long-running loops.
</instructions>

<examples>
## Bad
```go
func handler(w http.ResponseWriter, r *http.Request) {
    ctx, _ := context.WithTimeout(r.Context(), 5*time.Second) // cancel leaked
    go func() {
        bgCtx := context.Background() // ignores request context
        doWork(bgCtx)
    }()
}
```

## Good
```go
func handler(w http.ResponseWriter, r *http.Request) {
    ctx, cancel := context.WithTimeout(r.Context(), 5*time.Second)
    defer cancel()
    go doWork(ctx) // propagate request context
}
```
</examples>

<patterns>
- `context.WithTimeout`/`WithCancel` where cancel is never stored or deferred
- Goroutines spawned from HTTP handlers using `context.Background()` instead of request context
- Long-running loops with blocking I/O that never check `ctx.Done()`
</patterns>
