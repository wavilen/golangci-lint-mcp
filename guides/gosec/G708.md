# G708

<instructions>
Detects Server-Side Template Injection (SSTI) where tainted user input flows into template execution. Go's `text/template` and `html/template` engines allow calling methods and accessing fields on objects, which can be exploited to read files, execute commands, or access internal application state.

Never pass user-controlled strings as template content. Templates should be defined by the application, not by user input. If dynamic templates are unavoidable, use a sandboxed template engine with restricted access to data and functions.
</instructions>

<examples>
## Bad
```go
import "text/template"

func render(userTemplate string, data any) (string, error) {
    // Attacker input: "{{.System.Exec \"id\"}}" or similar
    // depending on what methods are available on the data object
    tmpl, err := template.New("dynamic").Parse(userTemplate)
    if err != nil {
        return "", err
    }
    var buf bytes.Buffer
    err = tmpl.Execute(&buf, data)
    return buf.String(), err
}
```

## Good
```go
import "html/template"

// Define templates server-side; user input is only data
var safeTemplates = template.Must(template.New("page").Parse(`
<h1>{{.Title}}</h1>
<p>{{.Body}}</p>
`))

func render(title, body string) (string, error) {
    var buf bytes.Buffer
    err := safeTemplates.Execute(&buf, map[string]string{
        "Title": title,
        "Body":  body,
    })
    return buf.String(), err
}
```
</examples>

<patterns>
- Never pass user input to `template.Parse` — use static templates with data binding only
- Disable raw template editing in CMS features — use structured data binding instead
- Replace user-defined templates with pre-built templates — pass user data as template variables only
- Validate configuration-sourced template content against an allowlist of approved templates
</patterns>
