# staticcheck: SA9008

<instructions>
Detects `return` statements inside deferred functions. A `return` inside a deferred function suppresses the panic recovery mechanism and does not return a value to the caller. It also makes the control flow confusing — the deferred function's return value is discarded.

Avoid `return` in deferred functions. If you need to recover from a panic, use `recover()` and set named return values. If you need to modify the return value, use named returns.
</instructions>

<examples>
## Bad
```go
func safeRun() (err error) {
    defer func() {
        if r := recover(); r != nil {
            return // does not set err, confusing control flow
        }
    }()
    panic("oops")
    return nil
}
```

## Good
```go
func safeRun() (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("panic: %v", r)
        }
    }()
    doWork()
    return nil
}
```
</examples>

<patterns>
- `return` inside `defer func()` blocks
- `return` in deferred functions suppressing panics silently
- Confusing control flow where deferred return overrides function return
- Named return values modified incorrectly in deferred functions
</patterns>
