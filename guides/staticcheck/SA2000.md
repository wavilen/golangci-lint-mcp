# staticcheck: SA2000

<instructions>
Detects `sync.WaitGroup.Add` called inside the goroutine being waited on instead of before launching it. Calling `Add` inside the goroutine creates a race: the `WaitGroup` counter may still be zero when `Wait` is called, causing `Wait` to return prematurely before the goroutine starts work.

Call `wg.Add(1)` before the `go` statement, not inside the goroutine function.
</instructions>

<examples>
## Bad
```go
var wg sync.WaitGroup
for _, item := range items {
    go func(it string) {
        wg.Add(1)   // too late — Wait may already have returned
        defer wg.Done()
        process(it)
    }(item)
}
wg.Wait()
```

## Good
```go
var wg sync.WaitGroup
for _, item := range items {
    wg.Add(1) // before launching goroutine
    go func(it string) {
        defer wg.Done()
        process(it)
    }(item)
}
wg.Wait()
```
</examples>

<patterns>
- `wg.Add(1)` as the first line inside the goroutine function
- Assuming `wg.Add` and `go func` are interchangeable in ordering
- Patterns where the loop body has `go func() { wg.Add(1); ... }()`
</patterns>
