# staticcheck: SA6002

<instructions>
Detects `sync.Pool.Put` calls where the argument is a pointer to a value allocated on the heap just for the pool. This defeats the purpose of `sync.Pool`, which is to reuse allocations. Allocating a new pointer, putting it in the pool, and then allocating again on `Get` provides no benefit.

Store the actual object (not a new pointer wrapper) in the pool, or let `Get` return the reusable object directly.
</instructions>

<examples>
## Good
```go
var bufPool = sync.Pool{
    New: func() any {
        return &bytes.Buffer{}
    },
}

func process() {
    buf, _ := bufPool.Get().(*bytes.Buffer)
    defer func() {
        buf.Reset()
        bufPool.Put(buf)
    }()
    _ = buf
}
```
</examples>

<patterns>
- Remove unnecessary pointer indirection when storing objects in `sync.Pool`
- Reset objects before putting them back in the pool with `Put`
- Reuse objects from `Get` instead of allocating new ones in `Put`
- Avoid `sync.Pool` for small immutable objects that don't benefit from reuse
</patterns>
