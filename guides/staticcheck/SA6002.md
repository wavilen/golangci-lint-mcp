# staticcheck: SA6002

<instructions>
Detects `sync.Pool.Put` calls where the argument is a pointer to a value allocated on the heap just for the pool. This defeats the purpose of `sync.Pool`, which is to reuse allocations. Allocating a new pointer, putting it in the pool, and then allocating again on `Get` provides no benefit.

Store the actual object (not a new pointer wrapper) in the pool, or let `Get` return the reusable object directly.
</instructions>

<examples>
## Bad
```go
var bufPool = sync.Pool{
    New: func() interface{} {
        return new(bytes.Buffer) // allocates new pointer each time
    },
}

func getBuffer() *bytes.Buffer {
    return bufPool.Get().(*bytes.Buffer)
}
```

## Good
```go
var bufPool = sync.Pool{
    New: func() interface{} {
        return &bytes.Buffer{}
    },
}

func process() {
    buf := bufPool.Get().(*bytes.Buffer)
    defer func() {
        buf.Reset()
        bufPool.Put(buf)
    }()
}
```
</examples>

<patterns>
- Wrapping pool objects in unnecessary pointer indirection
- Not resetting objects before putting them back in pool
- Allocating inside `Put` instead of reusing from `Get`
- Pool used for types that don't benefit from reuse (small immutable objects)
</patterns>
